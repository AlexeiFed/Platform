# План реализации варианта B для марафонов

## Цель
Перевести марафон из модели "это почти обычный курс с `startDate` и набором уроков" в модель "это программа с календарём событий, индивидуальными процедурами и отдельным студентским сценарием".

Итог после реализации:
- марафон хранит длительность и нормальный жизненный цикл;
- контентные сущности отделены от календаря марафона;
- процедуры ведутся по индивидуальному графику участника;
- админ может собрать марафон как расписание, а не как плоский список уроков;
- студент видит не просто уроки, а план по дням и неделям.

## Почему меняем текущую модель
Сейчас `Product(type=MARATHON)` отличается от курса только `startDate`, а вся логика держится на `Lesson`, `unlockRule` и `unlockDay`.

Это не покрывает реальные сценарии:
- в один день должно быть несколько активностей;
- тренировки идут по разным трекам: дом и зал;
- процедуры не являются "уроками", у них индивидуальный график и факт прохождения;
- у марафона есть недели, этапы, финал и служебные активности;
- каталог сейчас скрывает марафон через сутки после старта, что ломает жизненный цикл продукта.

## Целевая предметная модель

### 1. Марафон как продукт с временными рамками
Нужно расширить `Product`:
- оставить `type = MARATHON`;
- добавить `endDate` или `durationDays`;
- использовать это в видимости марафона, доступности событий и UI.

Рекомендация:
- добавить `durationDays Int?`;
- вычислять `endDate` на уровне приложения при необходимости;
- для конкретного запуска этого марафона использовать `startDate = 2026-02-01`, `durationDays = 22`.

Почему `durationDays`, а не только `endDate`:
- проще считать `dayOffset`;
- легче тиражировать одинаковые марафоны на новые потоки;
- меньше ручных ошибок в админке.

### 2. Календарь событий марафона
Нужна новая сущность, условно `MarathonEvent`.

Назначение:
- хранить структуру марафона по дням;
- показывать несколько сущностей в один день;
- отделить календарь от учебного контента.

Пример полей:

```prisma
enum MarathonEventType {
  INFO
  TRAINING
  NUTRITION
  PROCEDURE
  BONUS
  LIVE
  RESULT
}

enum MarathonTrack {
  ALL
  HOME
  GYM
}

model MarathonEvent {
  id          String            @id @default(uuid())
  productId   String
  title       String
  description String?           @db.Text
  type        MarathonEventType
  track       MarathonTrack     @default(ALL)

  dayOffset   Int
  weekNumber  Int?
  position    Int               @default(0)

  lessonId    String?
  blocks      Json?

  published   Boolean           @default(false)
  createdAt   DateTime          @default(now())
  updatedAt   DateTime          @updatedAt

  product     Product           @relation(fields: [productId], references: [id], onDelete: Cascade)
  lesson      Lesson?           @relation(fields: [lessonId], references: [id], onDelete: SetNull)

  @@index([productId, dayOffset, position])
  @@index([productId, type])
  @@map("marathon_events")
}
```

Что важно:
- `dayOffset` задаёт день относительно старта;
- `position` нужен для порядка внутри дня;
- `track` закрывает сценарий "дом / зал";
- `lessonId` позволяет переиспользовать текущую модель уроков;
- `blocks` даёт возможность делать событие без отдельного урока.

### 3. Индивидуальные процедуры участника
Процедуры нельзя хранить как обычные уроки. Нужен отдельный слой для участника.

Новые сущности:

```prisma
model ProcedureType {
  id        String   @id @default(uuid())
  title     String   @unique
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  items UserMarathonProcedure[]

  @@map("procedure_types")
}

model UserMarathonProcedure {
  id              String   @id @default(uuid())
  enrollmentId    String
  procedureTypeId String
  scheduledAt     DateTime?
  completedAt     DateTime?
  notes           String?  @db.Text
  position        Int      @default(0)
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt

  enrollment    Enrollment   @relation(fields: [enrollmentId], references: [id], onDelete: Cascade)
  procedureType ProcedureType @relation(fields: [procedureTypeId], references: [id], onDelete: Restrict)

  @@index([enrollmentId, position])
  @@index([procedureTypeId])
  @@map("user_marathon_procedures")
}
```

Эта модель нужна для:
- индивидуального графика;
- отметки факта прохождения;
- расчёта прогресса по процедурам;
- отображения "35 процедур из 35".

## Как раскладывается ваш марафон в новую модель

### Подготовительный этап
События `MarathonEvent` с `type=INFO`, `dayOffset=0`:
- доступ в закрытый чат;
- организационная информация;
- оплата;
- анкета участника.

Это не обязательно уроки. Можно сделать как `blocks`.

### Недели 1–3
Каждый день недели хранится через `dayOffset`.

Пример:
- понедельник: `TRAINING`, track `HOME`;
- понедельник: `TRAINING`, track `GYM`;
- вторник: `NUTRITION`, track `ALL`;
- пятница: `TRAINING`, track `HOME`;
- пятница: `TRAINING`, track `GYM`;
- дополнительные материалы: `BONUS`, track `ALL`.

### Процедуры
Есть два слоя:
- шаблонные события недели: `MarathonEvent(type=PROCEDURE)` как описание того, какие процедуры входят в неделю;
- конкретные записи участника: `UserMarathonProcedure` как его персональный график.

То есть в марафоне можно показать "на этой неделе идут LPG, роликовый массаж, прессотерапия", а в карточке участника уже держать конкретные даты и факт посещения.

### Финал
Отдельные события:
- `LIVE` для эфира;
- `RESULT` для разбора результатов;
- `BONUS` или `INFO` для рекомендаций после марафона.

## План внедрения по этапам

## Этап 1. Prisma и миграция данных
Что делаем:
- добавить в `prisma/schema.prisma` новые enum и модели;
- расширить `Product` полем `durationDays Int?`;
- при необходимости добавить связь `Product -> MarathonEvent`;
- не ломать текущие `Lesson` и `Enrollment`.

Что проверить:
- индексы на `productId`, `dayOffset`, `enrollmentId`;
- каскадное удаление для событий марафона;
- корректность типов для Prisma Client.

Результат этапа:
- база умеет хранить марафон как расписание и процедуры как индивидуальный план.

## Этап 2. Серверная доменная логика
Нужно добавить server actions, лучше в новой зоне:
- `src/app/(admin)/admin/courses/[courseId]/marathon-actions.ts`

Минимальный набор:
- `createMarathonEvent`
- `updateMarathonEvent`
- `deleteMarathonEvent`
- `reorderMarathonEvents`
- `createProcedureType`
- `assignProcedureToEnrollment`
- `updateUserProcedure`
- `deleteUserProcedure`

Требования:
- валидация через Zod;
- `try/catch`;
- проверка роли;
- `revalidatePath()` для админских и студентских страниц;
- единый контракт ответа `{ success, data?, error? }`.

## Этап 3. Исправление жизненного цикла марафона
Нужно переделать `src/lib/product-visibility.ts`.

Что меняем:
- убрать логику "скрыть через сутки после старта";
- для марафона ориентироваться на:
  - `published`;
  - `startDate`;
  - `durationDays`;
  - при необходимости на явный статус потока в будущем.

Практичный вариант сейчас:
- продукт виден до окончания марафона;
- если нужен добор участников, это потом выносится в отдельное правило продаж, а не в скрытие продукта.

## Этап 4. Admin UI для расписания марафона
Точка входа:
- `src/app/(admin)/admin/courses/[courseId]/course-editor.tsx`

Что добавить:
- отдельный таб или секцию "Расписание марафона";
- список дней/недель;
- форма создания события;
- выбор:
  - типа события;
  - дня марафона;
  - трека `ALL / HOME / GYM`;
  - режима контента:
    - привязать существующий `Lesson`;
    - задать `blocks` прямо в событии.

Дополнительно:

- быстрый дублирующий сценарий для Пн/Ср/Пт тренировок;
- шаблонный генератор недель, чтобы не забивать всё руками.

Важно:
- старый редактор уроков не удалять;

## Этап 6. Страница конкретного события

### Путь B

- `src/app/(student)/learn/[courseSlug]/event/[eventId]/page.tsx`

Рекомендация:

## Этап 7. Процедуры в админке и у студента

### Для админа

Подходящая зона:

### Для студента

Если само назначение делает админ, студенту пока хватит readonly-экрана.

## Этап 8. Пересчёт прогресса
Сейчас `Enrollment.progress` у вас общий и, вероятно, завязан на уроки.

Для марафона надо определить отдельную формулу:
- события с контентом;
- выполненные процедуры;
- при необходимости домашки.

Практичный старт:
- 70% веса = завершённые контентные события;
- 30% веса = завершённые процедуры.

Лучше не зашивать расчёт в SQL. Сделать отдельный серверный helper, чтобы формулу потом можно было менять.

## Что не нужно ломать в первой итерации
На первом проходе лучше не трогать:
- текущую модель `Lesson`;
- текущую домашку;
- текущий рендер урока;
- текущие курсы.

Новая логика должна включаться только для `Product.type === "MARATHON"`.

## Риски и решения

### Риск 1. Слишком много контента уйдёт в `blocks`
Решение:
- основные обучающие материалы держать в `Lesson`;
- `blocks` в `MarathonEvent` использовать для коротких инфо-событий.

### Риск 2. Дублирование данных между `Lesson` и `MarathonEvent`
Решение:
- считать `MarathonEvent` календарным слоем;
- основной контент хранить в `Lesson`;
- в событии хранить только календарную мету и ссылку на урок.

### Риск 3. Процедуры привязаны к участнику, а не к продукту
Это не баг, а правильная модель.
У марафона есть обещание "35 процедур в программе", но расписание и факт прохождения всегда принадлежат конкретному `Enrollment`.

## Конкретный результат первой рабочей версии
